# 运动商品 2022 年销售数据分析

## 项目背景

本项目以某运动品牌 2022 年全年销售数据为分析对象，运用 Python（pandas + matplotlib）完成数据读取、清洗合并、统计分析与可视化，旨在从月度趋势、品类结构等维度揭示销售规律，为经营决策提供数据支持。

## 数据说明

- **数据来源**：`商品销售数据.xlsx`
- **数据规模**：信息表 39 条商品 × 销售数据表 2040 条订单
- **时间跨度**：2022-01-01 至 2022-12-31
- **主要字段**：
  - 信息表：商品大类、商品小类、商品名称、商品编号、商品销售价
  - 销售数据表：订单日期、订单编号、商品编号、订单数量

## 分析方法

| 步骤 | 方法 | 说明 |
|------|------|------|
| 数据读取 | `pd.read_excel()` | 分别读取"信息表"和"销售数据表"两个工作表 |
| 数据合并 | `merge(on='商品编号')` | 按商品编号将信息表与销售表关联 |
| 月度统计 | `df['月份'].dt.month` + `groupby().sum()` | 提取月份、计算每月销售金额 |
| 可视化 | `matplotlib.pyplot` | 折线图、分组/水平柱状图、饼图、热力图 |

## 主要结论

1. **全年销售总额**：¥5,280,308
2. **品类结构**：
   - 运动上装贡献最大，占比 **62.2%**（¥3,282,383）
   - 运动下装次之，占比 **27.9%**（¥1,475,314）
   - 运动配饰为补充品类，占比 **9.9%**（¥522,611）
3. **月度趋势**：销售额呈季节性波动，年中（7 月）为最高峰（¥490,090），1 月为低谷（¥344,878）
4. **小类排名**：外套（¥982,788）> 背心 > 短袖 > 长袖 > 长裤 > 短裤 > 袜子 > 发带 > 手环
5. **数据质量**：销售表中 `cs002` 在信息表中无对应记录，已在内联时自动排除

## 个人反思：技术的未来与个体定位

### 所见：大模型的真实效率

本次分析体验让我重新审视了大语言模型在数据处理领域的实际能力——从数据读取到可视化，整个流程的代码生成几乎无缝。pandas 的 `merge` + `groupby` 这样的经典范式，大模型已经能精准调用；matplotlib 的中文渲染、图表美化也都一次性通过。这让我意识到，**"写代码"本身正在加速贬值**——如果一个任务能用清晰的逻辑描述出来，大模型大概率能给出可运行的实现。

### 所感：工具的精确位置

但与此同时，我注意到几个关键的"人的位置"：

1. **问题定义权**：模型不会主动判断"应该分析什么"——大类和商品划分、时间颗粒度、对比维度，这些决策仍依赖人的业务理解
2. **审美判断力**：图表配色、标注选择、信息密度，这些是 taste 问题而非技术问题
3. **异常发现力**：`cs002` 缺失这件事情，模型能检测到，但"要不要处理、如何处理、要不要溯源"的决策链条依然在人手里

换句话说，大模型正在从"编程工具"变成"编程能力的廉价放大器"，但它放大的前提是你得有一个**清晰的分析框架**塞进去。

### 所望：技术分工的重新想象

我越来越相信，未来的技术工作不会消失，但会剧烈分化：

- **执行层**（写 SQL、画图、调 API）将被大模型高度自动化
- **设计层**（定义分析问题、设计指标体系、判断异常是否有业务意义）将变得更加稀缺
- **整合层**（把分析结果嵌入业务决策流程、推动行动）的地位将上升

这种分工让我作为一个学习者的定位更加清晰：**不是去跟大模型比谁更会写代码，而是去培养它无法替代的能力——定义好问题、识别好答案、承担决策后果。**

本次项目虽然只是一个"合并两张表 + 画几张图"的简单流程，但它验证了一个假设：**在清晰指令下，大模型确实能显著压缩从数据到洞察的时间成本**。而"给出什么样的清晰指令"本身，或许就是这个时代最重要的元技能。
